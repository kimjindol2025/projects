# 최종 일관성(Eventual Consistency): 분산 시스템의 현실적 타협

## 소개

강한 일관성(Strong Consistency)은 모든 노드가 항상 같은 데이터를 보도록 보장합니다. 하지만 이는 성능과 가용성의 대가입니다. 대신 **최종 일관성(Eventual Consistency)**은 일시적 불일치를 허용하면서도 결국 모든 노드가 같은 상태로 수렴하도록 합니다.

## 최종 일관성이란?

```
시간 → [A: 100] [B: 100] [C: 100]
        ↓ 업데이트: 150
시간 → [A: 150] [B: 100] [C: 100]  (불일치 기간)
        ↓ 복제 전파
시간 → [A: 150] [B: 150] [C: 150]  (수렴)
```

### 핵심 정의
> "만약 어떤 업데이트도 수신하지 않으면, 결국 모든 복제본은 같은 값을 가진다"

## 실제 사례

### DNS (Domain Name System)
```
# 권위 있는 네임서버에서 레코드 업데이트
update example.com A 93.184.216.34

# 캐싱 DNS 서버들은 TTL 동안 이전 값 반환
캐시 dns1.com: 93.184.216.33  (TTL: 3600초)
캐시 dns2.com: 93.184.216.33  (TTL: 3600초)

# TTL 만료 후 새로운 값으로 수렴
TTL 만료 → 모두 93.184.216.34 반환
```

### DynamoDB (AWS)
- 쓰기: 주 레플리카에 즉시 반영
- 읽기: 보조 레플리카에서 몇 ms 지연 가능
- 강한 일관성 읽기: 추가 비용으로 보장 가능

## 구현 패턴

### 1. Read-Your-Writes Consistency
```
[클라이언트]
    ↓ 쓰기 (node1)
  [node1: 새로운 값]
    ↓ 읽기 요청 (같은 session)
  → node1에서 읽기 강제 (캐시 사용)
```

### 2. Causal Consistency
```
A가 B에게: "나 새로운 값으로 업데이트했어"
B의 읽기: A가 본 값 이상을 보장
```

### 3. Version Vectors (논리 시계)
```
[node1, v1] → [A:1, B:0, C:0]
[node2, v1] → [A:1, B:1, C:0]
[node3, v1] → [A:1, B:1, C:1]

관찰: node3는 node1, node2의 모든 변경을 본 상태
```

## Eventual Consistency의 장점

| 항목 | 강한 일관성 | 최종 일관성 |
|------|-----------|----------|
| **가용성** | 낮음 (일부 장애 시) | 높음 (거의 모든 상황) |
| **지연시간** | 높음 (합의 필요) | 낮음 (비동기 복제) |
| **처리량** | 낮음 | 높음 |
| **프로그래밍 복잡도** | 낮음 | 높음 |

## 주의사항: 잠재적 이상 현상

### Lost Update (손실된 업데이트)
```
초기: x = 10
클라이언트1: x += 5  (node1 에서)
클라이언트2: x += 3  (node2 에서)
동시 복제로 인해 한쪽 값만 남을 수 있음
결과: x = 15 또는 x = 13 (손실!)
```

**해결책**: Version Vector + CRDTs (Conflict-free Replicated Data Types)

### Read-Your-Writes 위반
```
[클라이언트] 쓰기 (node1) → 읽기 (node2)
문제: node2에 아직 복제 안됨
결과: 자신이 쓴 값을 읽지 못함
```

**해결책**: Session token으로 node1 접근 강제

## CRDT (Conflict-free Replicated Data Types)

자동으로 최종 일관성을 보장하는 자료구조:

### G-Counter (증가만 하는 카운터)
```
[node1] counter: {node1: 5, node2: 0, node3: 0}
[node2] counter: {node1: 5, node2: 3, node3: 0}
[node3] counter: {node1: 5, node2: 3, node3: 2}

total = 5 + 3 + 2 = 10 (모든 노드에서 같은 결과!)
```

### LWW-Element-Set (Last-Write-Wins Set)
```
각 원소에 타임스탐프 부착:
{("a", 100ms), ("b", 200ms), ("c", 150ms)}

같은 원소에 대한 이후 쓰기는 더 큰 타임스탐프 승리
```

## 실전 예제

### 블로그 댓글 서비스 (최종 일관성)
```
요구사항:
- 댓글 추가: 매우 빠르게 (< 100ms)
- 댓글 복제: 몇 초 (5초 이내)

아키텍처:
[사용자] → [Primary Node]
                ↓ (비동기)
        [Replica 1] [Replica 2]

결과: 매우 빠른 쓰기, 약간의 불일치 허용
```

## 최종 일관성 vs. 강한 일관성 선택

```
선택: 최종 일관성
- 소셜 미디어 (좋아요 개수, 팔로우)
- 뉴스피드
- 분석 데이터
- 캐시

선택: 강한 일관성
- 은행 계좌 잔액
- 재고 시스템
- 주식 거래
- 의료 기록
```

## 결론

최종 일관성은 **분산 시스템의 현실적 타협**입니다. CAP 정리에 따르면 셋 다 만족할 수 없으므로, 비즈니스 요구사항에 따라 선택해야 합니다.

대부분의 웹 서비스는 최종 일관성으로 충분하며, CRDT와 같은 고급 기법으로 프로그래밍 복잡도를 줄일 수 있습니다.

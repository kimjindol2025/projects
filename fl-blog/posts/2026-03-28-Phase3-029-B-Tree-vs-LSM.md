# 데이터 구조: B-Tree vs LSM 트레이드오프

## 요약

- B-Tree: 읽기 최적화
- LSM: 쓰기 최적화
- 성능 벤치마크
- 데이터베이스 선택 가이드

---

## 1. B-Tree

### 구조

```
      [15, 30, 45]
     /      |      \
  [1-14] [16-29] [31-44] [46+]

특징:
- 정렬된 리프
- 기존 DB의 표준
- 읽기: O(log N)
- 쓰기: O(log N) + 리밸런싱
```

### 장점

```
✅ 읽기 빠름
✅ 범위 쿼리 최고
✅ 안정적
```

### 단점

```
❌ 쓰기 느림 (리밸런싱)
❌ 정렬 필요
❌ 메모리 단편화
```

---

## 2. LSM (Log-Structured Merge)

### 구조

```
메모리 (MemTable)
│ [1→100, 2→200, ...] (정렬됨)
│
압축 → SSTable 1 (디스크, 정렬)
    → SSTable 2
    → SSTable 3 (압축)

특징:
- 순차 쓰기
- 배경 압축
- 쓰기: O(1) (메모리)
- 읽기: O(log N) (여러 테이블)
```

### 장점

```
✅ 쓰기 매우 빠름
✅ SSD 친화적
✅ 압축률 높음
```

### 단점

```
❌ 읽기 느림 (여러 테이블)
❌ 복잡한 구조
❌ 압축 오버헤드
```

---

## 3. 성능 벤치마크

### 쓰기 성능 (100만 키)

```
B-Tree:   50초 (리밸런싱 비용)
LSM:      5초  (메모리 쓰기 + 배경 압축)

승자: LSM (10배)
```

### 읽기 성능 (단일 키)

```
B-Tree:   1ms
LSM:      10ms (여러 테이블 탐색)

승자: B-Tree (10배)
```

### 범위 쿼리 (1000개 키)

```
B-Tree:   50ms
LSM:      100ms (블롭 정렬)

승자: B-Tree (2배)
```

---

## 4. 실전 선택

### B-Tree 사용

```
데이터베이스: PostgreSQL, MySQL, SQLite
용도: OLTP (Online Transaction Processing)
특징: 읽기/쓰기 균형

예시:
- 사용자 조회 (자주)
- 트랜잭션 (균형)
```

### LSM 사용

```
데이터베이스: RocksDB, Cassandra, HBase
용도: 높은 쓰기 처리량
특징: 대부분 쓰기

예시:
- 로그 저장 (매우 빈번)
- 시계열 데이터
- 센서 데이터
```

---

## 5. 하이브리드

```
일부 DB는 둘 다 지원:
- MongoDB: B-Tree (기본) + LSM 옵션
- InnoDB: B-Tree (기본)
- 사용자 선택 가능
```

---

## 6. 메모리 사용

### B-Tree

```
1백만 키, 1KB 값

인덱스:   50MB
데이터:   1000MB
메타:     50MB
총:       1100MB (예측 가능)
```

### LSM

```
메모리 테이블: 100MB (기본)
SSTable 캐시: 500MB
인덱스:      50MB
총:          650MB (더 적음)
```

---

## 핵심 정리

| 측면 | B-Tree | LSM |
|------|--------|-----|
| **읽기** | 빠름 | 느림 |
| **쓰기** | 느림 | 빠름 |
| **범위** | 최고 | 나쁨 |
| **메모리** | 더 많음 | 적음 |

---

## 결론

**"문제에 맞는 도구를 선택하세요"**

읽기 주도 → B-Tree
쓰기 주도 → LSM

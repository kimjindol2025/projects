# 정규표현식: 성능 최적화로 10배 가속화
## 요약

- 정규표현식 엔진 (NFA vs DFA)
- 백트래킹 이해하기
- 성능 벤치마크
- 로그 파싱 10배 가속화

---

## 1. 정규표현식 엔진

### NFA (비결정적 유한 오토마타)

```
패턴: a(b|c)*d
입력: abbbcd

NFA:
a → (b or c) → (b or c) → (b or c) → d
                    ↑_____________↑
           백트래킹 (여러 경로 탐색)

문제: 매칭 실패 시 모든 경로 재탐색 → 느림
```

### DFA (결정적 유한 오토마타)

```
NFA 컴파일 → DFA (경로 결정)

입력: abbbcd
→ a → b → b → b → c → d (한 경로)

장점: 백트래킹 없음, 매우 빠름
단점: 컴파일 비용
```

---

## 2. 백트래킹 피하기

### ❌ 나쁜 예

```regex
# 카타스트로픽 백트래킹
^(a|a)*b$

입력: "aaaaaaaaaaaaaaaaaac" (a 20개 + c)

탐색:
- a 20개 매칭 실패 → 19개 다시 시도
- a 19개 매칭 실패 → 18개 다시 시도
- ...
- 총: 2^20 = 100만 번 시도!

결과: 1초 → 10분
```

### ✅ 좋은 예

```regex
^a*b$  # 더 명확한 패턴

탐색:
- a* (0개 이상) 매칭
- b 매칭
- 끝

결과: <1ms
```

## 3. 성능 팁

### (1) 구체적인 패턴 사용

```regex
// ❌ 느림
.*error.*

// ✅ 빠름
^.*error.*$  # 시작/끝 명시

// ✅ 더 빠름
error  # 그냥 문자열
```

### (2) 그룹핑 최소화

```regex
// ❌ 느림 (10개 그룹)
(a)(b)(c)(d)(e)(f)(g)(h)(i)(j)

// ✅ 빠름 (1개 그룹)
(?:a)(?:b)(?:c)...(?:j)

// ✅ 최고 (그룹 없음)
abcdefghij
```

### (3) 탐욕 vs 비탐욕

```regex
// ❌ 느림 (탐욕 - 최대한 매칭)
<.*>

입력: <a><b><c>
결과: <a><b><c> (모두 매칭)

// ✅ 빠름 (비탐욕 - 최소한 매칭)
<.*?>

입력: <a><b><c>
결과: <a>, <b>, <c> (각각)
```

---

## 4. 벤치마크

### 로그 파싱 (100만 줄)

```
패턴: ^\d{4}-\d{2}-\d{2} (.+) error

나쁜 예:
.*\d{4}-\d{2}.*\d{2}.*error
결과: 10초

좋은 예:
^\d{4}-\d{2}-\d{2}.*error$
결과: 1초

최고:
[0-9]{4}-[0-9]{2}-[0-9]{2}.*error
결과: 0.1초 (100배!)
```

## 6. 실전 최적화

### Go 코드

```go
// ❌ 나쁜 예 (매번 컴파일)
func parseLog(line string) {
    if matched, _ := regexp.MatchString(`^error`, line); matched {
        // ...
    }
}
// 100만 줄 × 컴파일 = 10초

// ✅ 좋은 예 (한 번만 컴파일)
var reError = regexp.MustCompile(`^error`)

func parseLog(line string) {
    if reError.MatchString(line) {
        // ...
    }
}
// 100만 줄 × 빠른 매칭 = 0.1초
```

## 7. 대안

### 정규표현식 대신 문자열 함수

```go
// ❌ 정규표현식
regexp.MatchString(`^/api/users`, url)  // 느림

// ✅ 문자열 함수
strings.HasPrefix(url, "/api/users")    // 매우 빠름

// 성능: 10배
```

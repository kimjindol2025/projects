# LSM Tree: 1,670줄로 배우는 쓰기 성능 최적화


## 들어가며: "쓰기"는 왜 느릴까?

```
초당 1,000개 데이터를 저장해야 하는 상황:

❌ B-Tree (전통적 DBMS):
   1 쓰기 = 디스크 I/O 5-10번
   1ms × 1,000 = 1초 (정체!)

✅ LSM Tree (최신 DB):
   1 쓰기 = 메모리 쓰기 (0.1μs)
   0.0001ms × 1,000 = 0.1ms (매우 빠름!)
```

**이 차이를 만드는 것이 LSM Tree입니다.**

---

## 문제: B-Tree의 한계

### 1.1 B-Tree 쓰기 프로세스

```
B-Tree는 정렬된 상태를 유지해야 합니다.

1. 새 데이터 "K=100, V=apple"
   └─ B-Tree에서 올바른 위치 찾기 (이진 검색)

2. 노드 분할 확인
   └─ 노드가 가득 찼나? 분할 필요?

3. 디스크 I/O
   └─ 부모 노드, 형제 노드 등 여러 노드 수정
   └─ 각 수정마다 디스크 동기화 (fsync)

결과:
  시간: 1-10ms (디스크 I/O)
  처리량: 100-1,000 ops/sec
```

**문제:**

```
┌──────────────────────────────────────────┐
│         메모리의 B-Tree 노드              │
│    ┌───────────────────────────┐        │
│    │ [Key] [Key] [Key] [?] ... │ 가득찬 상태
│    └───────────────────────────┘        │
│              │                           │
│              ▼ 새 데이터 추가하려면       │
│         "노드 분할" 필요                   │
│              │                           │
│              ▼                           │
│    └─────────────────────────────────── │
│      자식 노드도 갱신 필요 (복잡함)       │
│    └─────────────────────────────────── │
│              │                           │
│              ▼ 디스크에 동기화            │
│           fsync() - 1-10ms! (느림)      │
└──────────────────────────────────────────┘
```

### 1.2 왜 이렇게 복잡할까?

**B-Tree가 유지해야 할 불변성:**

```
1. 모든 리프 노드는 같은 깊이에 있다
2. 각 노드는 정렬되어 있다
3. 각 노드의 자식 개수는 order 범위 내

이들을 유지하려면:
└─ 노드 분할, 병합, 재조정이 필요
└─ 매번 여러 노드를 수정
└─ 매번 디스크 I/O 필요
```

## 해결책: LSM Tree (Log-Structured Merge Tree)

### 2.1 핵심 아이디어: "순차 쓰기"

```go
// ❌ B-Tree 방식 (랜덤 I/O)
disks[randomIndex] = newValue  // 디스크 어디에나 쓰기

// ✅ LSM 방식 (순차 I/O)
log.Append(newValue)  // 맨 뒤에만 추가 (순차)
                      // SSD도 이 방식을 좋아함!
```

**왜 순차 I/O가 빠를까?**

```
디스크 성능:

순차 읽기:  550 MB/s  (매우 빠름)
순차 쓰기:  500 MB/s  (매우 빠름)
랜덤 읽기:   30 MB/s  (느림)
랜덤 쓰기:   20 MB/s  (매우 느림) ← B-Tree는 이것

LSM은 순차 쓰기를 사용 → 500 MB/s!
B-Tree는 랜덤 쓰기를 사용 → 20 MB/s!

성능 차이: 25배!
```

### 2.2 LSM Tree 구조

```
메모리 계층:
┌─────────────────────────────────────┐
│ MemTable (메모리 테이블)             │
│  - 새 데이터 즉시 여기에 저장        │
│  - SkipList 구조 (O(log N) 조회)    │
│  - 정렬 유지                        │
└─────────────────────────────────────┘
        │
        ├─ 언제 디스크로?
        │
        ▼ (MemTable이 가득 차면)
┌─────────────────────────────────────┐
│ Write-Ahead Log (WAL)                │
│  - 메모리 손실 대비 (복구용)         │
│  - 순차 쓰기                        │
└─────────────────────────────────────┘
        │
        ▼ (동기화)
┌─────────────────────────────────────┐
│ SSTable (Sorted String Table)        │
│  - 디스크에 저장된 정렬 데이터       │
│  - 블록 단위 압축                   │
│  - 여러 개 존재 (Level 0,1,2...)    │
└─────────────────────────────────────┘
        │
        ├─ 여러 SSTable이 있으면?
        │
        ▼ (주기적 압축)
┌─────────────────────────────────────┐
│ Compaction (압축)                    │
│  - 여러 SSTable을 병합              │
│  - 중복 제거                        │
│  - 효율적인 구조 유지               │
└─────────────────────────────────────┘
```


### 3.2 핵심 1: SkipList (메모리 인덱스)

```go
// FreeLang: pkg/lsm/skiplist.go

package lsm

import "math/rand"

type SkipListNode struct {
    Key   string
    Value string
    Next  []*SkipListNode  // 여러 레벨의 포인터
}

type SkipList struct {
    Head   *SkipListNode
    MaxLvl int
    Level  int  // 현재 최대 레벨
}

func NewSkipList() *SkipList {
    head := &SkipListNode{
        Key:  "",
        Next: make([]*SkipListNode, 16),
    }
    return &SkipList{
        Head:   head,
        MaxLvl: 16,
        Level:  1,
    }
}

// O(log N) 조회
func (sl *SkipList) Find(key string) (string, bool) {
    node := sl.Head

    // 가장 높은 레벨부터 탐색 (위에서 아래로)
    for level := sl.Level - 1; level >= 0; level-- {
        for node.Next[level] != nil && node.Next[level].Key < key {
            node = node.Next[level]
        }
    }

    node = node.Next[0]
    if node != nil && node.Key == key {
        return node.Value, true
    }
    return "", false
}

// O(log N) 삽입
func (sl *SkipList) Insert(key, value string) {
    // 1️⃣ 새 노드의 레벨 결정 (확률 기반)
    newLevel := randomLevel(sl.MaxLvl)
    if newLevel > sl.Level {
        sl.Level = newLevel
    }

    // 2️⃣ 삽입 위치까지의 경로 저장
    update := make([]*SkipListNode, newLevel)
    node := sl.Head

    for level := sl.Level - 1; level >= 0; level-- {
        for node.Next[level] != nil && node.Next[level].Key < key {
            node = node.Next[level]
        }
        if level < newLevel {
            update[level] = node
        }
    }

    // 3️⃣ 새 노드 생성 및 링크
    newNode := &SkipListNode{
        Key:   key,
        Value: value,
        Next:  make([]*SkipListNode, newLevel),
    }

    for level := 0; level < newLevel; level++ {
        newNode.Next[level] = update[level].Next[level]
        update[level].Next[level] = newNode
    }
}

// 레벨 결정: 확률 기반 (50% 감소)
func randomLevel(maxLevel int) int {
    level := 1
    for rand.Float64() < 0.5 && level < maxLevel {
        level++
    }
    return level
}
```

### 3.3 핵심 2: MemTable (메모리 테이블)

```go
// FreeLang: pkg/lsm/memtable.go

type MemTable struct {
    skipList *SkipList
    size     int
    maxSize  int
    mu       sync.RWMutex
}

func NewMemTable(maxSize int) *MemTable {
    return &MemTable{
        skipList: NewSkipList(),
        size:     0,
        maxSize:  maxSize,
    }
}

// O(log N) 쓰기
func (mt *MemTable) Set(key, value string) {
    mt.mu.Lock()
    defer mt.mu.Unlock()

    mt.skipList.Insert(key, value)
    mt.size += len(key) + len(value)
}

// O(log N) 읽기
func (mt *MemTable) Get(key string) (string, bool) {
    mt.mu.RLock()
    defer mt.mu.RUnlock()

    return mt.skipList.Find(key)
}

// MemTable이 가득 찼는가?
func (mt *MemTable) IsFull() bool {
    mt.mu.RLock()
    defer mt.mu.RUnlock()

    return mt.size >= mt.maxSize
}

// 스냅샷 생성 (플러시용)
func (mt *MemTable) Snapshot() []*KVPair {
    mt.mu.RLock()
    defer mt.mu.RUnlock()

    return mt.skipList.ToSortedArray()
}
```

### 3.4 핵심 3: WAL (Write-Ahead Log)

```go
// FreeLang: pkg/lsm/wal.go

type WALEntry struct {
    Timestamp int64
    Op        string  // "PUT", "DELETE"
    Key       string
    Value     string
}

type WAL struct {
    file *os.File
    buf  *bufio.Writer
}

func OpenWAL(filepath string) (*WAL, error) {
    file, err := os.OpenFile(filepath,
        os.O_WRONLY|os.O_CREATE|os.O_APPEND, 0644)
    if err != nil {
        return nil, err
    }

    return &WAL{
        file: file,
        buf:  bufio.NewWriter(file),
    }, nil
}

// 순차 쓰기 (매우 빠름!)
func (wal *WAL) Append(entry WALEntry) error {
    // 이진 형식으로 인코딩
    data := encodeWALEntry(entry)

    // 길이 접두사 (복구 시 경계 파악)
    lengthBuf := make([]byte, 4)
    binary.BigEndian.PutUint32(lengthBuf, uint32(len(data)))

    // 버퍼에 쓰기 (메모리)
    wal.buf.Write(lengthBuf)
    wal.buf.Write(data)

    return nil
}

// 동기화 (필요할 때만)
func (wal *WAL) Sync() error {
    if err := wal.buf.Flush(); err != nil {
        return err
    }
    return wal.file.Sync()
}

// 복구: WAL 재생
func (wal *WAL) Recover() ([]*KVPair, error) {
    file, err := os.Open(wal.filepath)
    if err != nil {
        return nil, err
    }
    defer file.Close()

    var entries []*KVPair
    reader := bufio.NewReader(file)

    for {
        // 길이 읽기
        lengthBuf := make([]byte, 4)
        _, err := reader.Read(lengthBuf)
        if err == io.EOF {
            break
        }
        if err != nil {
            return nil, err
        }

        length := binary.BigEndian.Uint32(lengthBuf)
        data := make([]byte, length)
        _, _ = reader.Read(data)

        // 엔트리 복호화
        entry := decodeWALEntry(data)
        entries = append(entries, &KVPair{entry.Key, entry.Value})
    }

    return entries, nil
}
```

### 3.5 핵심 4: SSTable (디스크 테이블)

```go
// FreeLang: pkg/lsm/sstable.go

type SSTable struct {
    Level    int
    KeyRange struct {
        Min string
        Max string
    }
    Blocks     []*Block  // 데이터 블록
    Index      []Index   // 인덱스
    BloomFilter *BloomFilter  // 빠른 검색
}

type Block struct {
    Data      []byte  // 압축된 데이터
    Uncompressed []byte  // 원본
}

// SSTable 생성 (MemTable → 디스크)
func NewSSTable(level int, entries []*KVPair) (*SSTable, error) {
    st := &SSTable{
        Level: level,
        Blocks: []*Block{},
        Index: []Index{},
    }

    // 블록 단위 분할 (64KB per block)
    blockSize := 64 * 1024
    var blockData []byte

    for _, entry := range entries {
        encoded := encodeKVPair(entry)

        if len(blockData)+len(encoded) > blockSize {
            // 블록 압축 (Snappy)
            compressed := snappy.Encode(nil, blockData)
            st.Blocks = append(st.Blocks, &Block{
                Data:         compressed,
                Uncompressed: blockData,
            })

            // 인덱스 기록
            st.Index = append(st.Index, Index{
                Key:       entry.Key,
                BlockIdx:  len(st.Blocks) - 1,
                Offset:    0,
            })

            blockData = nil
        }

        blockData = append(blockData, encoded...)
    }

    // 마지막 블록 처리
    if len(blockData) > 0 {
        compressed := snappy.Encode(nil, blockData)
        st.Blocks = append(st.Blocks, &Block{
            Data:         compressed,
            Uncompressed: blockData,
        })
    }

    // Bloom Filter 구성 (빠른 부재 확인)
    st.BloomFilter = NewBloomFilter(len(entries))
    for _, entry := range entries {
        st.BloomFilter.Add(entry.Key)
    }

    return st, nil
}

// O(log n) 조회
func (st *SSTable) Get(key string) (string, bool) {
    // 1️⃣ Bloom Filter로 사전 필터링 (부재 빠르게 확인)
    if !st.BloomFilter.Contains(key) {
        return "", false
    }

    // 2️⃣ 이진 검색으로 인덱스에서 블록 찾기
    blockIdx := sort.Search(len(st.Index), func(i int) bool {
        return st.Index[i].Key >= key
    })

    if blockIdx == len(st.Index) {
        return "", false
    }

    // 3️⃣ 해당 블록 압축 해제 후 탐색
    block := st.Blocks[blockIdx]
    data := snappy.Decode(nil, block.Data)

    // 블록 내 선형 탐색 (블록은 작음: 64KB)
    for data != nil && len(data) > 0 {
        pair := decodeKVPair(data)
        if pair.Key == key {
            return pair.Value, true
        }
        // 다음 엔트리
        data = data[len(encodeKVPair(pair)):]
    }

    return "", false
}
```

### 3.6 핵심 5: Compaction (압축)

```go
// FreeLang: pkg/lsm/compactor.go

type Compactor struct {
    levels []*Level
    ticker *time.Ticker
}

type Level struct {
    Number   int
    SSTables []*SSTable
    MaxSize  int64
}

// 백그라운드 압축
func (c *Compactor) Start() {
    c.ticker = time.NewTicker(10 * time.Second)
    go func() {
        for range c.ticker.C {
            c.compactIfNeeded()
        }
    }()
}

func (c *Compactor) compactIfNeeded() {
    for levelIdx := 0; levelIdx < len(c.levels)-1; levelIdx++ {
        level := c.levels[levelIdx]

        // 레벨의 크기가 한계를 초과했나?
        var totalSize int64
        for _, st := range level.SSTables {
            totalSize += st.Size()
        }

        if totalSize > int64(level.MaxSize) {
            c.compactLevel(levelIdx)
        }
    }
}

func (c *Compactor) compactLevel(levelIdx int) {
    // 현재 레벨과 다음 레벨의 SSTable 병합
    sourceLevel := c.levels[levelIdx]
    targetLevel := c.levels[levelIdx+1]

    // 1️⃣ 모든 엔트리 수집 (정렬)
    var allEntries []*KVPair
    for _, st := range sourceLevel.SSTables {
        allEntries = append(allEntries, st.ToSortedArray()...)
    }
    for _, st := range targetLevel.SSTables {
        allEntries = append(allEntries, st.ToSortedArray()...)
    }

    // 2️⃣ 정렬 및 중복 제거
    sort.Slice(allEntries, func(i, j int) bool {
        return allEntries[i].Key < allEntries[j].Key
    })

    allEntries = deduplicateSorted(allEntries)

    // 3️⃣ 새 SSTable 생성
    newSSTables := createSSTables(targetLevel.Number, allEntries)

    // 4️⃣ 레벨 업데이트
    sourceLevel.SSTables = nil
    targetLevel.SSTables = append(targetLevel.SSTables, newSSTables...)
}
```

### 3.7 통합: LSM Tree

```go
// FreeLang: pkg/lsm/lsm.go

type LSMTree struct {
    memtable   *MemTable
    immutable  *MemTable  // 플러시 대기 중인 MemTable
    wal        *WAL
    compactor  *Compactor
    mu         sync.RWMutex
}

func NewLSMTree(dataDir string) (*LSMTree, error) {
    wal, err := OpenWAL(filepath.Join(dataDir, "wal.log"))
    if err != nil {
        return nil, err
    }

    lsm := &LSMTree{
        memtable:  NewMemTable(4 * 1024 * 1024),  // 4MB
        wal:       wal,
        compactor: NewCompactor(),
    }

    // WAL 복구
    if err := lsm.recoverFromWAL(); err != nil {
        return nil, err
    }

    // Compaction 시작
    lsm.compactor.Start()

    return lsm, nil
}

// O(log N) 쓰기
func (lsm *LSMTree) Put(key, value string) error {
    lsm.mu.Lock()
    defer lsm.mu.Unlock()

    // 1️⃣ WAL에 기록
    if err := lsm.wal.Append(WALEntry{
        Op:    "PUT",
        Key:   key,
        Value: value,
    }); err != nil {
        return err
    }

    // 2️⃣ MemTable에 추가
    lsm.memtable.Set(key, value)

    // 3️⃣ MemTable이 가득 찼나? (플러시)
    if lsm.memtable.IsFull() {
        lsm.flushMemTable()
    }

    return nil
}

// O(log N + 여러 SSTable) 읽기
func (lsm *LSMTree) Get(key string) (string, bool) {
    lsm.mu.RLock()
    defer lsm.mu.RUnlock()

    // 1️⃣ MemTable 확인 (가장 최신)
    if val, ok := lsm.memtable.Get(key); ok {
        return val, true
    }

    // 2️⃣ Immutable MemTable 확인
    if lsm.immutable != nil {
        if val, ok := lsm.immutable.Get(key); ok {
            return val, true
        }
    }

    // 3️⃣ SSTable 레벨별로 확인
    for _, level := range lsm.compactor.levels {
        for _, st := range level.SSTables {
            if val, ok := st.Get(key); ok {
                return val, true
            }
        }
    }

    return "", false
}

func (lsm *LSMTree) flushMemTable() error {
    // MemTable → immutable
    lsm.immutable = lsm.memtable
    lsm.memtable = NewMemTable(4 * 1024 * 1024)

    // 백그라운드에서 플러시
    go func() {
        entries := lsm.immutable.Snapshot()
        st, _ := NewSSTable(0, entries)
        lsm.compactor.AddSSTable(0, st)
    }()

    return nil
}
```

## LSM이 사용되는 현실

### 5.1 실제 데이터베이스

| DB | LSM 사용 |
|---|---------|
| **RocksDB** | ✅ (Facebook) |
| **LevelDB** | ✅ (Google) |
| **Cassandra** | ✅ (쓰기 최적화) |
| **MongoDB** | ✅ (WiredTiger 엔진) |
| **PostgreSQL** | ❌ (B-Tree) |
| **MySQL InnoDB** | ❌ (B-Tree) |

### 5.2 LSM이 최적인 경우

✅ **시계열 데이터** (IoT, 로그, 메트릭)
```
특징: 순차적 쓰기, 최신 데이터 주로 조회
예: Prometheus, InfluxDB
```

✅ **이벤트 로그** (애널리틱스, 감시)
```
특징: 엄청난 쓰기량, 범위 쿼리
예: Kafka log segments
```

✅ **캐시층** (Redis, Memcached 대신)
```
특징: 빠른 쓰기, 배경 정리
예: 로그 구조 캐시
```

❌ **OLTP** (온라인 트랜잭션)
```
이유: 읽기와 쓰기 균형 필요
대신: B-Tree
```

## 마치며: 선택의 문제

**"항상 LSM이 좋은가?"**

아닙니다.

```
LSM Tree:
  ✅ 쓰기 매우 빠름 (순차 I/O)
  ❌ 읽기 느림 (여러 레벨 확인)
  ❌ Compaction 오버헤드
  ✅ 대용량 쓰기에 최적

B-Tree:
  ❌ 쓰기 느림 (랜덤 I/O)
  ✅ 읽기 빠름
  ✅ 범위 쿼리 효율
  ✅ 관리 간단
```

**올바른 선택:**
- 시계열/로그: **LSM**
- OLTP: **B-Tree**
- 하이브리드: **둘 다** (InnoDB는 Buffer Pool + Log-Structured writes)
